    1. User Setup¶

    The ‘Timeline’ page is where all of the active firefighting and incident response will take place when using the product. Right away, download the mobile app (IOS App / Android App).

    1. First, set up your user profile, contact methods and paging policy by clicking on your username in the top right and then ‘Your Profile’.
    2. On this page you can manage user information, contact methods and individual paging policies (notification preferences).
    3. Under your ‘Primary Paging Policy’, set up a policy which includes a push notification in the first step. See the above knowledge base article for more information.

    1. Team Scheduling & Escalations¶

    Teams in VictorOps segregate functional groups with different administration of schedules and escalations. Administration rights can be restricted to individual users by designating a team admin, though read access is permitted globally for all users.

    1. From the ‘Teams’ page, create a new team and by clicking ‘Add User’ from the team’s page, invite a few users to the new team.

    2. Next, click on the Rotations tab and create a new rotation. A rotation is a container of one or more shifts. A shift is a repeating pattern of on-duty times which leverage user lineups to the order of on-duty responsibilities.

    There are limitless possibilities to the schedules that can be created. To walk through a common example schedule try building one from this guide. You will need a rotation in order proceed to the next section Rotations are only a component of on-call. The next tab on the team page, Escalation Policies, determines the order of escalation from the first level up. Following the linked guide, create an escalation policy with a few steps, referencing the rotation you just created in the first step.

    The last step before setting up integrations is to configure a routing key for your escalation policy. The routing key is a url-friendly name that maps alerts (coming from integrations with Splunk, SFx, REST, etc). For example, I could set a routing key named “database” and map it to an escalation policy on the Database team. One routing key can map to multiple escalation policies. Create a routing key that maps to the escalation policy you just created.

    3. Key Integrations¶

    VictorOps depends on integrations to ingest alerts, remember that it is not a monitoring tool. In order for VictorOps to be of use, it needs to be connected to a monitoring tool. The Integrations page lists all pre-built integrations, and within each integration is a linked guide to configuring the integration.

    As you might expect, VictorOps has special integrations within the Splunk family Splunk Integrations Phantom Integration SignalFx Integration

    For customers using other tools, browse the integration list, we have guide for integrating with most tools. If not, there also generic integration points for both email and REST. Rest Endpoint Email Endpoint

    Certain integrations are of particular importance for incident response. Mainly, these are integrations with chat or ITSM solutions. Follow the below guides as examples. Slack Integration SNOW Integration

    4. Rules Engine¶

    The Alert Rules Engine is a powerful tool that allows you to deliver helpful information and useful resources to your users at the same moment they are notified of the problem, as well as customize the behavior of your incidents. It is, essentially, a rules engine that allows you to set certain conditions, and trigger custom actions when those conditions are met. Getting paged for a non-critical issue? You can use the Alert Rules Engine to make sure that nobody gets paged for alerts that are not actionable. Want to attach remediation documentation directly to notifications so your users can immediately start solving the problem without having to dig around? The Alert Rules Engine allows you to attach image links and other URLs directly to your notifications.     5. Webhooks¶

    Outgoing webhooks provide a means to interact with 3rd party API’s upon specified triggering events in VictorOps. For example, sending a web-request to a chat tool like Microsoft Teams when a new incident is triggered in VictorOps.

    Based off of the list of events you can customize the request body and use variable expansion (the same syntax as with the rules engine) to dynamically reference fields within the context of the triggering event.

    6. Splunk-VictorOps Dashboard¶

    Using Splunk, live and historic reports and dashboards can be produced by ingesting VictorOps data into Splunk. These dashboards can be a great overview on the current state of systems and how people are responding to them. Several pre-built dashboards can be shared with customers (soon to be directly packaged with the VictorOps for Splunk app) but can all be easily customized to expose key metrics for each organization.     For Splunk cloud customers, the best route is to send data from VictorOps via webhooks and ingest the data using a HEC. For on prem deployments, polling the API (thus eliminating exposing systems to inbound web traffic) is the best route.
